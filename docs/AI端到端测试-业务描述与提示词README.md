# 向 AI 描述业务逻辑：端到端测试提示词指南

本文档说明：**在 AI 辅助测试与自动化设计场景下，你的核心产出是「把业务规则讲清楚」**，而不是亲自逐行写断言代码。文内提供：

- **提示词母版（填空版）**：任意功能可套用；
- **输出质量约束**：可贴在任何提示词末尾；
- **两条一体化场景提示词**：登录失败锁定（虚构业务）与 **vibe_proving**（与 `app/ui` + `app/tests/e2e/test_frontend_e2e.py` 选型一致）。

---

## 1. 为什么「讲清楚业务」优先于「自己写代码」

- **AI 擅长**：在规则明确时，批量生成用例步骤、表格、边界条件清单，并映射到 UI/API 断言思路。
- **人必须做对的事**：限定范围、给出准确的状态机/阈值/时间、说明与真实界面一致的元素与接口、指定输出形状（表格/编号列表/Gherkin 等）。
- **常见失败原因**：业务口述模糊（「正常登录」未定义）、忽略锁定/重试/并发、未绑定真实控件 id 或路由，导致 AI 写出「看似专业但不可执行」的用例。

**结论**：把提示词当成「给高级测试架构师的需求文档」来写；代码（Playwright/pytest 等）可由 AI 或你在第二步再落地。

---

## 2. 提示词五要素（建议顺序固定）

| 顺序 | 要素 | 你要写什么 |
|------|------|------------|
| ① | **角色与任务** | 角色头衔 + 具体交付物（如「设计 E2E 步骤，可交给自动化工程师实现」）。 |
| ② | **业务细节** | 主路径、分支、计数器、时间窗、与后端/会话的一致性；禁止含糊词。 |
| ③ | **输出格式** | 强制表格列名、是否要有前置条件/步骤编号、是否附 API 期望等。 |
| ④ | **边缘与异常** | 明确要求「除主路径外至少 N 条」并举例类（网络、连点、时钟回拨等）。 |
| ⑤ | **Few-shot** | 1～2 个「示范用例」行，锁定**风格与颗粒度**（列粒度、断言语气、测试数据写法）。 |

## 3. 提示词母版（填空复制）

将下方整段复制到对话中，把 `【】` 内替换为你的真实信息即可。

```markdown
## 角色与任务
你是一位高级测试架构师。请为【功能名称 / 用户故事编号】设计**端到端测试步骤**，交付物需满足：
- 可被自动化工程师翻译为 Playwright / Cypress / Selenium 步骤；
- 每一步绑定**可定位的 UI 选择器**或**可观测的网络请求**（若仅有文案无 id，请标注「建议开发补充 data-testid」）。

## 被测系统上下文（事实约束，禁止臆造）
- 入口 URL：【如 https://host/ui/】
- 环境：【dev / staging / 本地 mock】
- 相关前端文件或页面：【路径或组件名】
- 相关 API（方法 + 路径）：【列举】
- 关键 DOM（id / role / data 属性）：【列举】

## 业务规则（状态机与数据）
【用编号列表写清：前置条件、主路径、分支、计数器、时间窗、重置条件、与后端会话/cookie 的关系】

## 输出格式（必须遵守）
1. **前置条件表**（可选）：数据准备、账号角色、Mock 开关。
2. **主路径用例表**：Markdown 表格，列固定为 **用例ID | UI 元素 | 预期交互 | 断言结果 | 测试数据**。
3. **边缘/异常用例表**：同上表头；至少 **【N】** 条，编号 E01…。
4. **网络断言建议**：对每个关键请求给出 method、path 模式、期望状态码、可选 JSON 字段断言。
5. **风险与不可自动化项**：Flaky 点、需真实第三方、需时钟控制的用例单独列出。

## 边缘与异常（你必须主动挖掘）
除主路径外，请重点考虑并覆盖（可扩展）：【如：断网、超时、连点、双标签并发、会话过期、服务端 4xx/5xx、非法输入、越权】。

## Few-shot（请严格效仿下列颗粒度）
| 用例ID | UI 元素 | 预期交互 | 断言结果 | 测试数据 |
|--------|---------|----------|----------|----------|
| S01 | 【#selector】 | 【单击一次】 | 【可见【文案或子选择器】】 | 【键值】 |
| S02 | 【#selector】 | 【输入并失焦】 | 【无错误态 / 特定校验文案】 | 【键值】 |
```

## 4. 追加条款：输出质量约束（可贴在任意提示词末尾）

```markdown
## 附加约束
- 所有用例必须有唯一 **用例ID**（主路径 Sxx，异常 Exx）。
- **断言结果**必须可判定真假：避免「界面正常」类模糊句；优先「URL / 可见选择器 / 网络请求次数 / 响应字段」。
- 若信息不足，请输出 **「待产品确认」** 清单，而不是猜测业务规则。
- 表格中 **UI 元素**列优先写 `id` 或 `role+name`；若用 XPath 仅作备选并说明脆弱性。
- 涉及时间的断言（如锁定 15 分钟）须说明 **如何在不真实等待 15 分钟的情况下测**（Mock 时钟、调短配置、直接注入服务端状态等）。
```

## 5. Few-shot 颗粒度参考（虚构登录，仅示范风格）

| 用例ID | UI 元素 | 预期交互 | 断言结果 | 测试数据 |
|--------|---------|----------|----------|----------|
| S01 | `#username` | 输入后失焦 | 无红色校验错误 | 用户：`demo_user` |
| S02 | `#btn-login` | 单击一次 | 出现 `#dashboard` 且 URL 含 `/home` | 密码：`Correct#1` |

## 6. 一体化场景提示词 A：登录失败锁定（直接复制给 AI）

```markdown
## 角色与任务
你是一位高级测试架构师，请为以下**用户登录**功能设计详细的端到端测试步骤（可落地为 Playwright / Cypress / Selenium）。

## 业务规则
- 登录失败累计 **3 次**后，账号进入**锁定**状态，**15 分钟内**不允许再次成功建立会话（错误提示文案以产品为准，但**锁定前后行为须一致、可重复**）。
- 锁定计时结束后，失败次数**按产品说明重置**，允许重新尝试。
- **成功登录**后，失败计数应**清零**。
- 会话标识（cookie / token）在锁定期的期望行为请与「仍显示锁定提示、不颁发新会话」一致。

## 输出格式
请用 **Markdown 表格**输出，列必须包含：**用例ID | UI 元素 | 预期交互 | 断言结果 | 测试数据**。主路径与异常路径分表。

## 边缘与异常（至少 5 条，须落到表格行）
请重点思考并列出：模拟网络断连、提交按钮**连点**、锁定期内用**正确密码**重复提交、**时钟跳变**（接近解锁边界）、**多标签页**并发登录、服务端在第 2 次失败后返回**非预期状态码**等。

## Few-shot（请效仿颗粒度，勿更粗）
| 用例ID | UI 元素 | 预期交互 | 断言结果 | 测试数据 |
|--------|---------|----------|----------|----------|
| S01 | `#password` | 第 1 次提交错误密码 | 提示错误；失败计数=1；未锁定 | 用户：`u1`，密码：`Wrong1` |
| S02 | `#password` | 第 2 次提交错误密码 | 提示错误；失败计数=2；未锁定 | 密码：`Wrong2` |

## 附加交付
- 给出 **锁定态** 与 **解锁态** 下的网络请求/响应断言建议（路径模式 + 状态码）。
- 说明 **15 分钟** 窗口的 E2E 实施策略（真实等待 / Mock / 配置缩短），并标注推荐方案。
```

## 7. 一体化场景提示词 B：vibe_proving Web UI（与仓库前端 / E2E 对齐）

以下选择器与接口与当前仓库一致：`app/ui/index.html`、`app/ui/app.js`、`app/tests/e2e/test_frontend_e2e.py`。若你本地改过 id，以代码为准。

```markdown
## 角色与任务
你是一位高级测试架构师。请为 **vibe_proving** Web 客户端设计端到端测试步骤，覆盖下列**用户可见主路径**（可被 Playwright 直接翻译）：
1. 打开 **设置面板**，填写并保存 **LLM** 配置；
2. **学习模式**：输入问题并发送；
3. **问题求解模式**：输入并发送；
4. **证明审查模式**：文本输入与（可选）**附件**按钮可见性；
5. **定理检索模式**：输入查询并发送；
6. **语言切换**（设置内 `#lang-seg`，`button[data-lang='en'|'zh']`）对界面文案的影响（至少断言一处 `.panel-title` 或等价可见文案）。

## 被测系统上下文（事实）
- 入口：`http://localhost:8080/ui/`（或部署基址 + `/ui/`）。
- 顶栏模式 Tab：`button.mode-tab[data-mode='learning'|'solving'|'reviewing'|'searching'|'formalization']`。
- 主输入：`#input-textarea`；发送：`#send-btn`（部分流程亦可能出现 `#btn-submit`，以页面为准）。
- 设置：`#btn-panel-toggle` 打开；面板容器 `#settings-panel`；字段 `#input-llm-base-url`、`#input-llm-api-key`、`#input-llm-model`；保存 `#btn-save-llm`。
- 审查附件按钮：`#attach-btn`。
- 配置加载：`GET /config`；保存 LLM：`POST /config/llm`；健康检查：`GET /health`；定理检索：`GET /search?q=...&top_k=10`。
- 学习 / 求解等：流式接口如 `POST /learn`、`POST /solve`（断言以「收到流式片段 / 结束事件 / UI 出现助手气泡」为主，勿虚构 SSE 字段名）。

## 业务规则
- 首次进入应请求配置；失败时 UI **不白屏**，配置区呈现降级状态（参考 `loadAppConfig` 行为）。
- 保存 LLM 须触发 `POST /config/llm`；成功后有可观察反馈（按钮状态、toast、徽章等，以实际 UI 为准）。
- 切换模式后，输入栏与发送逻辑保持一致；审查模式下附件能力与 `app.js` 中 `updateMode` 行为一致（非审查模式若有附件策略亦请注明）。
- 定理检索结果须 **有明确空/非空** 用户反馈，不得静默失败。

## 输出格式
1. **主路径表**：列 **用例ID | UI 元素 | 预期交互 | 断言结果 | 测试数据**（按模式分子表亦可）。
2. **Mock/桩建议表**：列 **HTTP 方法 | 路径模式 | 场景 | 建议状态码/体**。
3. **边缘/异常表**：至少 **5 条**，例如：`/config` 超时或 500、`/config/llm` 保存失败、`/search` 被 Abort、流式输出中途切换 Tab、未配置 API Key 时发起依赖 LLM 的模式、连点 `#send-btn` 导致重复请求等。

## Few-shot
| 用例ID | UI 元素 | 预期交互 | 断言结果 | 测试数据 |
|--------|---------|----------|----------|----------|
| S01 | `#btn-panel-toggle` | 单击 | `#settings-panel` 可见 | — |
| S02 | `button.mode-tab[data-mode='searching']` | 单击 | `aria-selected="true"` | — |

## 附加约束
- 与「追加条款：输出质量约束」一节一致：用例ID、可判定断言、时间敏感用例说明 Mock 策略。
- 若某步需真实大模型或外部 OCR，请标为 **E2E 环境分级**（smoke 跳过 / 夜间全量）。
```

## 8. 使用建议

- **先贴接口与 id**：从开发者工具复制 1～2 条真实请求样例（脱敏）贴进提示词，可显著减少 AI 臆造。
- **锁定「可观测」断言**：优先写 URL、网络请求、DOM 状态、可见文案；少写无法稳定的像素比对。
- **版本冻结**：注明「基于某 commit / 某日构建」，避免 UI 改版后用例漂移。
- **与仓库 E2E 对齐**：生成用例后可对照 `app/tests/e2e/test_frontend_e2e.py` 与 `test_e2e_robust.py` 补缺或合并重复。
