# 手动测试计划 - 数据流和用户交互验证

## 目标
验证数据流正确性、逻辑流程合理性、用户交互友好性

## 测试步骤

### 1. Solving模式 - 简单问题
**输入**：`求解方程：x² + 5x + 6 = 0`

**预期行为**：
- 输出应包含：x = -2, x = -3（或因式分解形式）
- 响应时间应在10-20秒内
- 进度提示清晰

**实际检查**：
- [ ] 输出内容是否与输入匹配？
- [ ] 是否有不相关内容（如Prime Number Theorem）？
- [ ] 响应时间是否合理？
- [ ] 用户能否理解当前进度？

### 2. Solving模式 - 复杂问题
**输入**：`证明：对于任意正整数n，存在n个连续的合数`

**预期行为**：
- 应输出构造性证明（如(n+1)!+2到(n+1)!+n+1）
- 可能需要30-60秒
- 应显示引用的定理

**实际检查**：
- [ ] 是否正确处理了复杂问题？
- [ ] TheoremSearch是否返回相关结果？
- [ ] 验证步骤是否有必要？

### 3. Reviewing模式
**输入**：sqrt(2)无理数证明

**预期行为**：
- 指出证明的逻辑步骤
- 标注正确/有gap的部分
- 响应时间20-30秒

**实际检查**：
- [ ] 是否正确识别了证明步骤？
- [ ] 验证结果是否合理？

### 4. 模式切换
**操作**：Learning → Solving → Reviewing → Searching

**实际检查**：
- [ ] 切换时是否清理了之前的状态？
- [ ] 输入框是否正确切换（Reviewing有上传按钮）？
- [ ] 历史记录是否正确保存？

## 发现的具体问题

### 数据流问题
- **问题1**：Solving模式输出不相关内容
  - **位置**：截图显示Prime Number Theorem，但输入是二次方程
  - **可能原因**：
    1. TheoremSearch缓存污染
    2. LLM模型幻觉
    3. 流式输出混乱
  - **如何复现**：提交简单二次方程，查看输出

### 逻辑流程问题
- **问题2**：简单问题过度复杂化
  - **位置**：solver.py的GVR循环
  - **建议**：添加简单问题检测，跳过验证步骤

### 用户交互问题
- **问题3**：进度反馈不足
  - **位置**：Solving/Reviewing模式长时间处理时
  - **建议**：
    - 显示预估剩余时间
    - 提供"取消"按钮
    - 超时后显示明确的错误提示
